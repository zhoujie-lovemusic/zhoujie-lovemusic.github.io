Async/Await

异步数据作为函数的返回值来使用

  • 想要获取类似接口这样的异步数据作为函数的返回值在别处使用，必须使用await/async，因为只有await才能把数据返回出去
  • 而async函数返回的一定是一个Promise，因此在别处使用又要用await
  • async一定还要加在离当前await最近的函数上，如果是回调函数要加在回调函数前面

想要一直保持同步，必须要一直连续await

  • 原来内部有await，然后调用外部函数，仍然需要再await才能保证这个函数 同步执行，并不是函数内部有await就行了

异步操作的遍历不能用map函数

  • 异步操作的遍历不能用map函数，否则返回的每一个item是一个promise，即使使用了await关键字也无法等待，因为map不回去处理异步操作，比较急躁

  • 要用for循环（item的某一个属性值单独用await，然后再赋值上去）或者用Promise.all所有的promise来解决（用了map之后，是一个包含promsie的数组，那么就用Promise.all获取所有结果）

布局

经验

  • 父元素来控制边距 和 子元素间距，子元素就直接full就行

flex布局

  • flex-wrap ———— 自动换行，只要是遍历某个元素，父元素都加
  • flex-1 ————– flex-grow（占比）、flex-shrink（收缩比例）、flex-basis（基础大小），如果只是单纯的剩余空间随父元素自适应，那么直接flex:1，如果各个子元素有相应占比需求，还是用flex-grow，因为直接写flex:1、flex:2在纵轴方向有时候占比会反着来
  • 只要想内容不会挤出来，想要自动撑开父元素 直接flex布局即可（不要加高度）—-> 子元素多大就占多大，不像div默认占一行
  • flex布局需要两边都设置宽度才不会相互挤压（只设置一边，另一边仍然会把这边给挤压变小、下次一边设置宽度，另一边直接用flex-1填充剩余（也是加了宽度））

————-上面两点用一句话总结：父元素加了flex子元素会’野蛮生长->弹性’，得加一点其他限制，才能更好控制：gap、flex-1、grow-1

flex和grid布局的各自使用场景

—>flex

  • 一维（一个方向的轴）
  • 各个元素不需要指定具体的宽高，需要利用flex的自适应

—>grid

  • 二维（横纵轴）
  • 指明了具体的列数和宽度/比例（网格）

grid的使用

  • 父元素通过grid-template-columns和grid-template-rows既能控制 水平还是垂直，还能控制每个元素之间的宽度/比例
  • 设置每列具体宽度或者使用repeat(个数,1fr)，1fr是每一行/列 自己平均分配 fraction 区域
  • repeat(个数,1fr)是可以和单个 多少fr用字符串拼接的，这样就可以 特殊处理某一列，然后剩余再用repeat拼接

flex布局和绝对定位的各自使用场景

  • flex是父元素控制子元素位置，而绝对定位是 自己控制自己的位置
  • 优先使用 flex，flex布局无法解决，然后再使用 绝对定位（如果有一些子元素有特殊的 距离控制，用padding更好）
  • 所以 flex->padding->绝对定位

Vue

引入组件一定要在setup函数里，因为这里面才可以默认注册

从外部引入的数据都要在当前组件重新赋一个变量来使用（解耦）

  • ref变量
  • 从父组件
  • 从store中

从store拿过来的数据（ref）不需要 .value（Vue3）,类似模板使用数据，特殊处理过了

在使用store的时候 不要使用computed（接父元素数据），而是usexxStore()—调用函数，计算属性只是获取某个变量而已和函数调用没关系

数据很复杂时，不知道什么时候.value，就记住一点：必须先.value之后才能正常取数据

defineProps没有用变量接受是不能在setup里面用的，只能在模板上用来展示

封装组件 用插槽可以 帮助你 传递自己的结构进来，就是这样来应用个性化场景的

  • 写slot不写名字，会默认有个名字 #default，其实就是默认插槽和具名插槽

涉及到响应式对象内部数据的修改（计算），要先深克隆到一个变量再处理

  • 因为类似直接赋值的浅拷贝本质山地址一样—->响应式更新还保留着，并且内部深层都有响应式

动态修改Html结构-v-html

  • 可以在原本要显示数据的地方使用v-html指令，数据用个计算属性（提供一个计算的平台）

  • 本质上是用html字符串的处理，那么就可以用各种字符串处理方法，不好处理用split转成数组处理，最后在拼接（join方法）

  • 找到文字内关键词并标绿

模态框（dialog）作为子组件显示，可见性需要更新回父组件

  • 子组件要使用计算属性接受，顺便利用set来更新回去

  • 不是单向数据流吗？为什么要更新回去？因为父子组件各自都有权力控制显示，不更新回去任一方的更新都将无效，且这样不用

  • this.$emit(‘update:exportFailVisible’, val);是用于更新用v-model绑定的数据，也就是说子组件是使用v-model绑定可见性的

父组件传数据给子组件而不是子组件每次实例化来请求数据

  • 这样一次请求就行了，而且子组件的展示也不是用v-if，而是用可见性

全局样式要去掉scoped

  • 有时候即使使用了deep穿透样式还是无法生效，就要F12看看是否是全局样式，也就是在body下的第一层，而一个组件内的样式加了scoped是指局部样式，所以要去掉scoped

watch和computed

  • 只用重新返回一个值的话用computed，否则有其他复杂逻辑用watch

计算属性传参计算需要返回一个回调函数在里面return才可以

  • computed设计初衷是根据实例已有的数据进行计算，而形参并不属于已有的数据，所以无法在computed回调函数里面去获得

计算属性写get个set的时候不需要写回调函数，里面只是要一个对象即可，因为get和set是键值类型的键

封装的组件拿来用一定要首字母大写，否则不算做一个组件，显示不出来

v-for和v-if不能一起用，那么就提前用计算属性把list筛选好

正向代理和反向代理的区别

  • 正向代理是代理客户端，服务器不用知道具体客户端是什么，只用知道代理客户端的地址
  • 反向代理是代理服务器，客户端不用知道具体服务器的地址，只用知道代理服务器的地址
  • 代理服务器可以进行转发多个目标，这样就有只知道一个地址却能一对多的效果

UI组件如果提供了专门的class自定义类，那肯定是可以用的，不生效是方法不对

el-upload

:beforfe-upload和:on-change的区别

  • beforfe-upload在上传之前就触发了，后面就都不会再触发了
  • on-change在文件上传、上传失败、上传成功都会触发
  • 如果只想在文件上传之前触发一次—->beforfe-upload；想阶段分的再细点—->on-change（用的更多一点）

手动触发action，而不是自动触发

  • :auto-upload:“false”
  • 注意在触发之前，千万不能this.refs.upload.clearFiles()，没有文件是不会触发action的
  • 利用ref手动触发：this.refs.upload.submit()

Git

删除分支（保持分支整洁）

  • 删除远程分支 —— git branch -D xxx
  • 删除本地分支 —— git branch -d xxx

新建远程分支

  • 只能用git push origin 分支名（没有会自动创建）

git add -A而不是git add .

  • git add -A还能提交文件删除的记录，而git add .不能

git pull –rebase而不是git pull

  • rebase可以在合并操作后不会留下commit记录，保证提交记录线性化，更清晰
  • 如果合并碰到冲突，使用git add+git rebase –continue完成剩余的合并操作

业务场景

无感刷新

  • 用setInterval每隔一秒调用获取列表的接口（记得组件销毁时clearInterval）———–刷新
  • “删除”、“编辑”等修改列表数据的操作时需要clearInterval，操作成功后再开启 —- 保持无感刷新
  • 把列表加载数据时的loading效果要去掉 ——— 无感

template-admin框架的下拉框独有的回显方式

  • 内置有useProxy方法获取某个下拉框数据的（字典值和label）组成的数组
  • 这个字典值会默认从store中去取，没有还会自动去请求接口并保存在store中
  • 如果是 修改 模式，则在具体的字典值来之前准备好数组即可；如果是 查看 模式，则还需要另外的方法将字典值转化为对应的label值

下载切图时一定要用svg格式的，避免分辨率不同带来的模糊

  • svg格式的图片能自适应还不失真

多个数据同时开定时器实现数据变化

  • 就是每一个定时器 用完就关，再检查一下，直到检查到最后一个定时器全部完成再提示

  • 开定时器最好再加个timeout，避免定时器相互影响

搜索框搜索的几种实现方式

  • 封装的useCommonTable 这个可以直接传入接口函数、搜索参数，就可以获取tableData
  • 也可以用watch和watcheffect（整个获取list接口写在里面，用到的数据变了就会变）

永远要用父组件控制自组件的的位置和大小

  • 子组件 长宽 全部100%即可
  • 父组件来控制大小和位置

涉及到列表的循环展示的，最外面要加el-scrollbar滚动条

文件导出（下载）一定要在接口返回值加上返回值类型，比如responseType: ‘blob’，一般都是 blob对象

工作经验

效率相关

  • 和后端联调时，直接先设计好数据结构然后对应的参数传上，后面通知后端按照这个来就行。这样就不会被后端该接口影响进度

  • 改Bug时，原有的东西能不动则不动，我没有必要花时间熟悉原来其他的逻辑

  • 找组件时，右键转到定义就可以直接去对应组件内部，不用自己去找 或者 control+单击

  • debug时，打印语句到当前模块开发完毕再统一删掉，因为可能整个过程都要用到

  • 解决问题时，非常偏没有实际案例的问题优先找AI（chatgpt）
